# 日志、评估和部署

> 大模型应用能跑起来只是第一步。真正上线后，你还要知道它慢在哪里、贵在哪里、错在哪里、什么时候变差。

---

## 0. 学完这一篇你会什么？

你会学会：

- 大模型应用应该记录哪些日志
- 如何做请求耗时统计
- 如何记录 Prompt 版本
- 如何做简单评估集
- 如何避免改 prompt 后效果变差
- 如何做限流、缓存和重试
- 如何部署 Node 大模型服务
- 如何控制成本

## 1. 为什么大模型应用更需要日志？

普通 API 错了，通常是：

```text
请求参数错
数据库错
业务逻辑错
```

大模型应用还会多出很多“不稳定因素”：

- 模型回答不稳定
- Prompt 改了效果变差
- 检索结果不对
- 输出格式不符合 JSON
- token 太多导致成本高
- 流式输出中途断开
- 用户输入诱导模型跑偏

所以从第一天就要记录日志。

---

## 2. 一次模型调用要记录什么？

建议记录：

```ts
type LlmRequestLog = {
  requestId: string;
  userId?: string;
  model: string;
  promptVersion: string;
  promptLength: number;
  outputLength?: number;
  latencyMs: number;
  status: "success" | "error";
  errorMessage?: string;
  createdAt: string;
};
```

如果是 RAG，还要记录：

```ts
type RagLog = {
  requestId: string;
  query: string;
  topK: number;
  sources: Array<{
    title: string;
    source: string;
    score: number;
    chunkIndex: number;
  }>;
};
```

为什么要记录 sources？

因为当回答错了，你要判断：

```text
是检索错了？
还是模型看到了正确资料但答错了？
```

## 3. Node 里记录耗时

最简单写法：

```ts
const startedAt = Date.now();

try {
  const answer = await generateText(input);

  console.log({
    requestId,
    model: config.hfModel,
    promptVersion: PROMPT_VERSION,
    latencyMs: Date.now() - startedAt,
    status: "success",
  });

  res.json({ answer });
} catch (error) {
  console.error({
    requestId,
    model: config.hfModel,
    latencyMs: Date.now() - startedAt,
    status: "error",
    errorMessage: getErrorMessage(error),
  });

  throw error;
}
```

后续可以换成 pino、winston 或接入日志平台。

---

## 4. requestId 很重要

每个请求生成一个 ID：

```ts
import crypto from "node:crypto";

export function createRequestId() {
  return crypto.randomUUID();
}
```

响应里也返回：

```ts
res.json({
  requestId,
  answer,
});
```

用户反馈“这次答错了”时，你可以通过 requestId 查到：

- 当时的 prompt
- 用的模型
- 检索来源
- 耗时
- 错误堆栈

## 5. 评估集是什么？

评估集就是一组固定测试题。

例如：

```json
[
  {
    "id": "node-hf-start",
    "question": "Node Hugging Face 示例怎么启动？",
    "expectedKeywords": ["examples/node-huggingface", "npm install", ".env", "npm run dev"]
  },
  {
    "id": "rag-meaning",
    "question": "RAG 是什么？",
    "expectedKeywords": ["检索", "生成", "参考资料"]
  }
]
```

每次你改 prompt、换模型、改检索逻辑，都跑一遍评估集。

目标不是让评估百分百自动判断，而是快速发现明显退化。

---

## 6. 最小评估脚本思路

```ts
for (const item of evalSet) {
  const answer = await ask(item.question);

  const hitCount = item.expectedKeywords.filter((keyword) =>
    answer.includes(keyword)
  ).length;

  console.log({
    id: item.id,
    hitCount,
    total: item.expectedKeywords.length,
    pass: hitCount >= Math.ceil(item.expectedKeywords.length * 0.7),
  });
}
```

这只是很粗糙的评估，但已经比完全靠感觉强很多。

以后可以升级：

- 人工打分
- LLM-as-judge
- 对比两个 prompt 版本
- 记录线上用户点赞/点踩

## 7. 限流

大模型接口贵，而且容易被刷。

最小限流策略：

```text
同一个 IP 每分钟最多 20 次
同一个用户每天最多 500 次
单次 maxNewTokens 不超过 1000
prompt 长度不超过 8000 字符
```

Node 可以用：

```text
express-rate-limit
```

也可以用 Redis 自己实现。

## 8. 缓存

什么时候适合缓存？

- FAQ
- 固定文档问答
- 相同 prompt 多次请求
- embedding 结果
- 文档切块结果

不适合缓存：

- 用户隐私数据
- 强实时数据
- 个性化强的回答
- 包含权限差异的结果

最适合先缓存的是 embedding：

```text
同一段文档内容 → embedding 不变
```

可以用内容 hash 做 key：

```ts
import crypto from "node:crypto";

export function hashText(text: string) {
  return crypto.createHash("sha256").update(text).digest("hex");
}
```

## 9. 超时和重试

模型服务可能慢或失败。

建议：

```text
普通生成：30-60 秒超时
流式生成：连接建立超时 + 空闲超时
embedding：10-30 秒超时
```

重试策略：

```text
网络错误：可以重试 1-2 次
参数错误：不要重试
鉴权错误：不要重试
模型输出格式错：可以让模型修复一次
```

注意：重试会增加成本。

---

## 10. 成本控制

大模型成本通常和 token 数、请求次数、模型价格有关。

控制方式：

- 限制 prompt 长度
- 限制 maxNewTokens
- RAG 只取 topK 相关片段
- 文档 chunk 不要过长
- 常见问题做缓存
- 简单任务用小模型
- 复杂任务才用大模型
- 记录用户级别用量

日志里建议记录：

```ts
type UsageLog = {
  userId: string;
  model: string;
  inputLength: number;
  outputLength: number;
  requestCount: number;
};
```

如果 SDK 能返回 token usage，就记录真实 token。不能返回时，先记录字符长度也有价值。

## 11. 部署结构

最小结构：

```text
浏览器
  ↓
Node API 服务
  ↓
Hugging Face / 其他模型服务
```

加 RAG 后：

```text
浏览器
  ↓
Node API 服务
  ↓
Postgres / Vector DB
  ↓
Hugging Face / 其他模型服务
```

生产环境还需要：

- HTTPS
- 环境变量管理
- 日志收集
- 错误报警
- 数据库备份
- 访问鉴权
- 限流

## 12. Docker 部署思路

示例 Dockerfile：

```dockerfile
FROM node:22-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

ENV NODE_ENV=production
EXPOSE 3001

CMD ["npm", "start"]
```

生产运行时传环境变量：

```bash
docker run \
  -p 3001:3001 \
  -e HF_TOKEN=hf_xxx \
  -e HF_MODEL=gpt2 \
  node-huggingface-demo
```

不要把真实 token 写进 Dockerfile。
