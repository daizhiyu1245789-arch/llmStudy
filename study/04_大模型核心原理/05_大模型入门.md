# 大模型入门

> 这一篇把前面学过的 Python、PyTorch、Tokenizer、Attention 串起来，帮你建立“大模型”的整体地图。

---

## 1. 这个项目在大模型学习里处于什么位置？

本项目 `tinyllm` 是一个极小版 GPT。它不是为了训练出 ChatGPT 那种效果，而是为了让你看懂大模型的完整骨架。

```
真实大模型：
海量文本 → 复杂 tokenizer → 超大 Transformer → 大规模预训练 → 指令微调 → 对齐 → 部署服务

本项目：
sample.txt → Char/Byte tokenizer → 小 GPT → 小规模训练 → 文本生成 → API 部署
```

对应关系：

| 工业大模型 | 本项目 tinyllm | 你能学到什么 |
|---|---|---|
| Tokenizer | `tinyllm/tokenizer.py` | 文本怎么变成 token ID |
| Transformer | `tinyllm/model.py` | GPT 的核心结构 |
| 预训练 | `tinyllm/train.py` | 怎么用下一个 token 训练模型 |
| Checkpoint | `tinyllm/checkpoint.py` | 怎么保存和恢复模型 |
| 推理生成 | `tinyllm/generate.py` | temperature、top-k 怎么影响输出 |
| 部署 | `study/05_大模型应用实战/03_部署成API服务.md` | 前端怎么调用模型 |

一句话：**tinyllm 是一个可以拆开看、可以跑起来、可以自己改的小型大模型实验台。**

## 2. 什么是大模型？

大模型通常指参数量很大、训练数据很大、能力比较通用的神经网络模型。

对语言模型来说，它的核心任务很朴素：

```
给定前文，预测下一个 token。
```

例如：

```
输入：今天晚上我想吃
预测：火锅 / 面条 / 米饭 / ...
```

训练时，模型不是直接“理解世界”，而是在海量文本里反复练习：

```
看到前 n 个 token → 猜第 n+1 个 token → 猜错了就调整参数
```

当模型足够大、数据足够多、训练足够充分时，它会学到很多语言规律、知识关联、代码模式和任务格式。于是我们就可以通过 prompt 让它完成总结、翻译、问答、写代码等任务。

## 3. 大模型的核心流水线

```
原始文本
  ↓
Tokenizer：文本切成 token ID
  ↓
Embedding：token ID 变成向量
  ↓
Transformer Block × N
  ↓
LM Head：输出每个 token 的概率
  ↓
采样策略：选出下一个 token
  ↓
Decode：token ID 变回文本
```

在本项目里：

| 流水线步骤 | 对应代码 |
|---|---|
| Tokenizer | `CharTokenizer` / `ByteTokenizer` |
| Embedding | `GPT.wte` 和 `GPT.wpe` |
| Transformer Block | `Block` |
| Attention | `CausalSelfAttention` |
| MLP | `MLP` |
| LM Head | `GPT.lm_head` |
| 生成 | `GPT.generate()` |

你读 `model.py` 的时候，可以带着这条流水线去看，不要一开始就陷在每一行张量维度里。

## 4. GPT 为什么叫“因果语言模型”？

GPT 的训练方式是从左到右预测下一个 token。

```
文本：今 天 天 气 很 好

训练样本：
输入 x: 今 天 天 气 很
目标 y: 天 天 气 很 好
```

模型在预测“气”的时候，只能看到：

```
今 天 天
```

不能偷看后面的：

```
很 好
```

这就是 `CausalSelfAttention` 里因果掩码的作用。

```
第 1 个位置：只能看第 1 个 token
第 2 个位置：能看第 1-2 个 token
第 3 个位置：能看第 1-3 个 token
...
```

所以 GPT 特别适合生成：每生成一个 token，就把它接到上下文后面，再继续预测下一个。

## 5. 参数量、数据量、算力分别影响什么？

大模型效果通常受三个因素影响：

| 因素 | 影响 | 在 tinyllm 里怎么调 |
|---|---|---|
| 参数量 | 模型容量，能记住和表达多少模式 | `--n_layer`、`--n_head`、`--n_embd` |
| 数据量 | 模型能学到多少语言和知识 | `--data_path` 换更大语料 |
| 训练步数 | 学习是否充分 | `--max_steps` |
| 上下文长度 | 一次能看多长文本 | `--block_size` |
| 采样策略 | 输出稳定还是发散 | `--temperature`、`--top_k` |

简单理解：

```
参数量太小：脑容量不够
数据太少：见识不够
训练太短：还没学会
上下文太短：记不住前面说了什么
采样太随机：容易胡说
```

本项目默认配置很小，适合学习流程。想要明显更好的生成效果，优先顺序通常是：

```
更干净的数据 → 更多训练步数 → 更大模型 → 更长上下文
```

## 6. 预训练、微调、对齐分别是什么？

### 6.1 预训练

让模型在海量文本上学习“预测下一个 token”。

```
目标：学语言、知识、代码模式、常见任务格式
```

本项目的训练就是一个极小版预训练。

### 6.2 指令微调

预训练模型只是会续写，不一定会听指令。指令微调会给模型看很多“用户问题 → 理想回答”的样本。

```
用户：帮我总结这段话
助手：...
```

目标是让模型学会对话格式和任务执行方式。

### 6.3 对齐

让模型更符合人的偏好，比如更有帮助、更安全、更少胡编。

常见方法包括：

```
人工偏好数据
奖励模型
RLHF / DPO 等训练方式
```

入门阶段不用急着实现这些，先把“预训练语言模型”这一层吃透。

## 7. 学大模型最容易混淆的概念

| 概念 | 简单解释 |
|---|---|
| Token | 模型处理文本的最小片段 |
| Embedding | token 的向量表示 |
| Context Window | 模型一次最多能看的 token 数 |
| Logits | Softmax 前的原始分数 |
| Loss | 模型预测错了多少 |
| Perplexity | 模型困惑程度，越低越好 |
| Temperature | 生成时的随机性 |
| Top-k | 只从概率最高的 k 个候选里采样 |
| Checkpoint | 模型训练到某一步的存档 |
| Fine-tuning | 在特定数据上继续训练 |
| Inference | 用训练好的模型生成结果 |

## 8. 从前端视角理解大模型应用

如果你是前端工程师，可以把大模型应用拆成三层：

```
UI 层：输入框、对话流、文件上传、结果展示
API 层：鉴权、限流、流式响应、日志、缓存
模型层：prompt、模型选择、RAG、工具调用、微调
```

常见产品形态：

| 产品形态 | 核心能力 |
|---|---|
| 聊天助手 | 多轮对话、上下文管理 |
| 文档问答 | RAG、向量检索、引用来源 |
| 代码助手 | 代码理解、补全、修改 |
| 智能客服 | 意图识别、知识库、转人工 |
| Agent | 工具调用、任务拆解、状态管理 |

学完 tinyllm 后，你可以继续往两个方向走：

```
方向 A：模型原理
Transformer → 训练优化 → 微调 → 对齐

方向 B：应用工程
Prompt → RAG → Tool Calling → Agent → 部署监控
```

## 9. 建议入门路线

### 第一轮：先看懂本项目

```
1. 跑通训练和生成
2. 看懂 tokenizer.py
3. 看懂 model.py 的 GPT、Block、Attention
4. 看懂 train.py 的训练循环
5. 改一次参数，观察 loss 和生成结果变化
```

### 第二轮：连接真实大模型

```
1. 学 Hugging Face 的 AutoTokenizer / AutoModel
2. 用一个开源模型做推理
3. 理解 prompt、system message、temperature
4. 做一个简单 API 服务
5. 前端接入流式输出
```

### 第三轮：进入应用开发

```
1. 做文档问答 RAG
2. 学向量数据库和 embedding
3. 学工具调用 function calling
4. 做一个能查资料、调用接口、生成结果的 Agent
5. 加上日志、评估、缓存、限流
```
