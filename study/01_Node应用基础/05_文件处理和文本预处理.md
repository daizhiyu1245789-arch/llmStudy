# 文件处理和文本预处理

> RAG 的第一步不是调用模型，而是把用户文档变成干净、可检索的文本。Node 开发者要先会读文件、清洗文本、切块。

---

## 0. 学完这一篇你会什么？

你会学会：

- Node 里怎么读文本文件
- 上传文件后要做哪些检查
- 文本清洗要清什么
- 怎么做基础切块
- chunk 里要保存哪些元数据
- 为什么文件处理要考虑安全

---

## 1. RAG 为什么需要文件处理？

RAG 入库链路：

```text
文件
  ↓
解析文本
  ↓
清洗文本
  ↓
切块
  ↓
生成 embedding
  ↓
存向量库
```

如果前面文本处理很差，后面 embedding 和检索都会受影响。

## 2. 读取文本文件

```ts
import fs from "node:fs/promises";

export async function readTextFile(filePath: string): Promise<string> {
  return fs.readFile(filePath, "utf8");
}
```

读取 Markdown：

```ts
const content = await readTextFile("README.md");
```

---

## 3. 上传文件要检查什么？

至少检查：

- 文件大小
- 文件类型
- 文件扩展名
- 用户权限
- 是否允许持久保存

示例规则：

```text
只允许 .txt / .md
单文件不超过 5MB
同一个用户最多上传 100 个文件
```

不要随便处理任意文件。

---

## 4. 文本清洗

基础清洗：

```ts
export function normalizeText(text: string): string {
  return text
    .replace(/\r\n/g, "\n")
    .replace(/\n{3,}/g, "\n\n")
    .replace(/[ \t]+/g, " ")
    .trim();
}
```

作用：

- 统一换行
- 去掉过多空行
- 压缩多余空格
- 去掉首尾空白

不要过度清洗。

如果把标题、列表、代码块都清掉，模型反而少了结构信息。

---

## 5. 基础切块

```ts
export function chunkText(text: string, chunkSize = 800, overlap = 120): string[] {
  const chunks: string[] = [];
  let start = 0;

  while (start < text.length) {
    const end = Math.min(start + chunkSize, text.length);
    const chunk = text.slice(start, end).trim();

    if (chunk) {
      chunks.push(chunk);
    }

    start += chunkSize - overlap;
  }

  return chunks;
}
```

参数建议：

```text
chunkSize: 500-1000 中文字符
overlap: 80-150 中文字符
```

---

## 6. Markdown 更适合按标题切

Markdown 有结构：

```md
# 大标题
## 小标题
正文...
```

简单按二级标题切：

```ts
export function splitMarkdownByHeading(markdown: string): string[] {
  return markdown
    .split(/\n(?=##\s+)/)
    .map((part) => part.trim())
    .filter(Boolean);
}
```

真实项目里可以组合：

```text
先按标题切
如果某块太长，再按长度切
```

## 7. chunk 元数据

不要只保存内容。

```ts
type Chunk = {
  id: string;
  documentId: string;
  title: string;
  source: string;
  content: string;
  chunkIndex: number;
};
```

为什么要元数据？

- 回答时展示引用来源
- 调试检索结果
- 删除文档时删除对应 chunks
- 重新入库时覆盖旧版本

---

## 8. 文件安全注意点

不要相信用户上传文件名。

用户可能上传：

```text
../../somewhere/secret.txt
```

所以保存文件时：

- 自己生成文件 ID
- 不用用户文件名当路径
- 限制上传目录
- 限制文件大小
- 不执行上传内容
